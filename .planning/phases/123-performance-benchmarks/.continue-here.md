## Two Runtime Bugs to Fix

### Bug 1: `meshc run` — unrecognized subcommand
`start-servers.sh` line 34: `meshc run "$REPO_ROOT/benchmarks/mesh/bench.mpl"`
meshc has NO `run` subcommand. Commands available: `build`, `init`, `repl`, `fmt`, `lsp`, `migrate`, `deps`.

**Fix needed in `start-servers.sh`:**
- `meshc build` expects a PROJECT DIRECTORY (with `main.mpl`), not a single file.
- `bench.mpl` is a standalone file in `benchmarks/mesh/bench.mpl`.
- Options:
  A. Create a project dir: `benchmarks/mesh/` already has `bench.mpl` — check if it has `main.mpl` or rename. Use `meshc build benchmarks/mesh/` to produce a binary, then run that binary.
  B. Check if `meshc` has a way to compile a single file (look at `Build` command args carefully).
  C. Use `meshc build` output binary path from the build command.

Look at `compiler/meshc/src/main.rs` for the Build command output path to know what binary to run after `meshc build`.

### Bug 2: Rust benchmark binary — Exec format error
Log: `/app/benchmarks/rust/target/release/bench: cannot execute binary file: Exec format error`

**Root cause investigation needed:**
- `.dockerignore` has `target/` — does this exclude `benchmarks/rust/target/`?
  - Docker `.dockerignore` typically matches `target/` at any depth (not just root)
  - If true, the macOS arm64 binary from local `benchmarks/rust/target/` is NOT copied
  - So the exec format error must come from the multi-stage COPY
- **Likely actual cause**: The COPY order in runtime stage:
  1. `COPY --from=builder /app/benchmarks/rust/target/release/bench .../bench` (Linux x86_64)
  2. `COPY benchmarks/ benchmarks/` — if `benchmarks/rust/target/` is NOT excluded by dockerignore,
     the macOS arm64 `bench` binary from local disk overwrites the builder's x86_64 binary.

**Fix options:**
  A. Move `COPY --from=builder .../bench` to AFTER `COPY benchmarks/ benchmarks/` in Dockerfile.
  B. Add `**/target/` to `.dockerignore` (stricter pattern).
  C. Both.

**Recommended fix:** Do both A and B to be safe.
